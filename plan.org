* Mission

A configurable simulator of a continuous intraday electricity exchange,
exposing the Frequenz Electricity Trading gRPC API. The reference market
behaviour is a continuous intraday electricity exchange for the DE-LU bidding
zone, but every market knob is data-driven so the same binary can stand
in for FR, AT, NL, BE, or any made-up zone a scenario calls for.

Same operating model as [[file:../switchyard][switchyard]]:

- Rust owns the deterministic machinery: order book, matching engine,
  state transitions, stream multiplexing, time.
- Lisp (tulisp) owns the world: which gridpools exist, what background
  liquidity looks like, how prices drift, what scenarios fire when. The
  config file is hot-reloadable and is the only place a user-visible
  knob lives.
- An axum-backed SPA shows the order book, trade tape, per-gridpool
  positions, scenario timeline, and a live REPL.
- A =tsctl= CLI mirrors =swctl=: invoke RPCs, run scenarios, dump
  state.

The simulator is for: testing trading-bot code without a real exchange
account; running deterministic ML-training replays; demoing
forecasting/strategy work; rehearsing operational playbooks for
gate-closure crunches.

** Why "tradingsim"?

Plain, descriptive, hard to mistake. The CLI binary is =tsctl=. The
crate name and the binary names are unambiguous.

- API spec: =submodules/frequenz-api-electricity-trading/proto/frequenz/api/electricity_trading/v1/electricity_trading.proto=

- Common types: =submodules/frequenz-api-electricity-trading/submodules/frequenz-api-common/proto/frequenz/api/common/v1alpha8/=

* The market model

** Delivery products

| Product | DeliveryDuration enum    | Start alignment              |
|---------|--------------------------|------------------------------|
| 5-min   | =DELIVERY_DURATION_5=    | multiples of 5 min past hour |
| 15-min  | =DELIVERY_DURATION_15=   | :00 :15 :30 :45              |
| 30-min  | =DELIVERY_DURATION_30=   | :00 :30                      |
| 60-min  | =DELIVERY_DURATION_60=   | :00                          |

Each contract = (delivery area, delivery period). Has its own order
book. =market::contract_id(area, period) -> ContractId= is the key.
v0.1 supports 60-min only; v0.3 expands.

** Delivery areas (initial supported set)

| EIC                | Symbol     | Currency |
|--------------------|------------|----------|
| 10Y1001A1001A82H   | DE_LU      | EUR      |
| 10YFR-RTE------C   | FR         | EUR      |
| 10YAT-APG------L   | AT         | EUR      |
| 10YNL----------L   | NL         | EUR      |
| 10YBE----------2   | BE         | EUR      |

Stored as data in =market.rs= so adding zones is config, not code. The
proto =DeliveryArea {string code, EnergyMarketCodeType code_type}=
maps to an internal =Area= newtype that the sim keeps interned.

** Tick + quantity conventions

| Param          | Default       |
|----------------|---------------|
| Price tick     | 0.01 EUR/MWh  |
| Min quantity   | 0.1 MW        |
| Quantity step  | 0.1 MW        |
| Price domain   | -9999 .. 9999 EUR/MWh |

All are per-area knobs in lisp (=(set-tick-size 'de-lu 0.01)=, etc.).
Pre-trade validation rejects out-of-grid prices/quantities and reports
=STATE_REASON_VALIDATION_FAIL=.

** Order types (proto =OrderType=)

| Enum             | v0.1 | v0.2 | v0.3 | Notes |
|------------------|------|------|------|-------|
| LIMIT            | ✓    | ✓    | ✓    | resting w/ optional valid_until |
| STOP_LIMIT       |      | ✓    | ✓    | activated when public last price crosses stop_price |
| ICEBERG          |      | ✓    | ✓    | display_quantity + peak_price_delta; new slice on slice fill |
| BLOCK            |      |      | v0.6 | linked periods, executed at common ratio |
| BALANCE          |      |      |      | deferred — not in proto MVP |
| PREARRANGED      |      |      |      | deferred |
| PRIVATE          |      |      |      | deferred |

** Execution restrictions (proto =OrderExecutionOption=)

| Enum  | Behaviour                                                                                  |
|-------|--------------------------------------------------------------------------------------------|
| (none)| stays on book until filled, cancelled, expired                                              |
| AON   | only matches if a counter-side aggregate exists ≥ full qty; never partially filled          |
| FOK   | match in full immediately or reject; never sits on book                                     |
| IOC   | match what you can immediately; cancel the rest; never sits on book                         |

** Order state machine (proto =OrderState=)

#+begin_example
                ┌──────────┐
   CreateOrder ─┤ PENDING  │── validation OK ─►┌────────┐
                └────┬─────┘                   │ ACTIVE │
                     │ validation fail          └───┬────┘
                     ▼                              │
                ┌────────┐    ┌──────────────────┬──┴────────┬───────────┐
                │ FAILED │    ▼                  ▼           ▼           ▼
                └────────┘ ┌──────┐         ┌────────┐  ┌─────────┐  ┌──────────┐
                           │FILLED│         │CANCELED│  │ EXPIRED │  │HIBERNATE │ (stop unarmed)
                           └──────┘         └────────┘  └─────────┘  └──────────┘
#+end_example

PENDING -> ACTIVE happens on the next match tick (one tick of book
admission). HIBERNATE is reserved for stop orders waiting for their
trigger. Every transition emits an =OrderDetail= update on the gridpool
stream and (if the order is publicly visible) a
=PublicOrderBookRecord= event.

** Trade state machine (proto =TradeState=)

v0.1 supports =ACTIVE= only. Cancel / recall workflows
(=CANCEL_REQUESTED=, =RECALLED=, =APPROVAL_REQUESTED=) come in v0.6+
once we have a "market operator" actor surface in lisp.

** Continuous matching algorithm

Per the continuous-matching algorithm spec, the bidyard^Wtradingsim matching
engine is:

1. Incoming order =O= validated (price tick, qty step, currency match,
   valid_until in future, area belongs to the gridpool, etc.).
2. If FOK/IOC/AON, attempt immediate sweep against the resting opposite
   half-book; FOK rolls back any partial fill if the full qty isn't
   reachable.
3. Otherwise, sweep against the resting half-book in price-time priority
   (best price first; among same price, oldest first).
4. Each match emits two private =Trade= rows (one per side's gridpool)
   plus one =PublicTrade= row visible globally.
5. If qty remains and =O= is restable, insert it at the price-time tail
   of its side and emit =PublicOrderBookRecord= with =STATE_REASON_ADD=.
6. Iceberg: only the =display_quantity= slice is on the book; on slice
   fill, the next slice is inserted at the current
   =price + peak_price_delta= (sell) or =price - peak_price_delta=
   (buy), with a =STATE_REASON_ICEBERG_SLICE_ADD=, losing time priority.
7. Order modify: any update that increases qty or improves price loses
   time priority (this matches intraday-market behaviour). Quantity decrease and
   tag/payload changes preserve priority.

Determinism: matching is single-threaded inside =World::tick=, so two
orders submitted in the same tick are ordered by their arrival sequence
(monotonic =u64= assigned at admission). Tests rely on this.

** Gate closure + delivery clock

A contract is "open for trading" between =gate_open_at(period)= and
=gate_close_at(period)=. Both are functions of (area, duration) and are
configurable from lisp; defaults match DE-LU as of 2026:

| Duration | Default gate open                  | Default gate close       |
|----------|------------------------------------|--------------------------|
| 60-min   | day-ahead 15:00 D-1 (CET)          | delivery start           |
| 30-min   | day-ahead 15:30 D-1                | delivery start           |
| 15-min   | 2 h before delivery hour           | delivery start           |
| 5-min    | 1 h before delivery 5-min slot     | delivery start           |

(For cross-border / SIDC traffic the gate closes 30 min before
delivery; v0.3 introduces a =cross-border= flag on the contract.)

At =gate_close_at=, all remaining ACTIVE orders on the contract are
forcibly transitioned to =EXPIRED= with =STATE_REASON_DELETE=,
=MARKET_ACTOR_SYSTEM=. Iceberg slices behind the visible one also
expire.

** Synthetic counterparties (the "environment animation" layer)

In switchyard, lisp animates voltage / consumer load / solar
irradiance; here lisp animates counterparties. A counterparty is a
synthetic trader that posts orders against the public book.

Counterparties are =SimulatedAgent= objects ticked once per
=physics_tick_ms=. Each one has a Lisp closure that decides what to
post on a given tick. Hooks Rust gives the closure:

- =(post-public-order …)=  - inserts an order from a non-gridpool source
- =(cancel-public-order id)= - cancels by id
- =(best-bid AREA PERIOD)=, =(best-ask AREA PERIOD)=, =(mid AREA PERIOD)=,
  =(last-trade AREA PERIOD)= - read-only book inspection
- =(qty-on-bid AREA PERIOD price)= / =(qty-on-ask …)= - depth at a price level
- =(seconds-to-gate-close AREA PERIOD)= - drives last-minute behaviour

A handful of canned strategies live in =sim/counterparties.lisp=
("market-maker", "spike-trader", "renewable-iceberg-seller",
"battery-arb"). User configs can drop them in by name or define their
own.

*** Demand / surplus tilt

The market-maker exposes two scalar knobs that shift quotes
independently:

- =demand= (EUR/MWh) — adds to the bid. A "demand" scenario procures
  aggressively, lifting the prevailing price. User sells against the
  MM clear faster.
- =surplus= (EUR/MWh) — subtracts from the ask. A "surplus" scenario
  dumps curtailment, depressing prices. User buys against the MM
  clear faster.

Both are positive numbers; they affect different sides so they can
fire together (a scenario where the market is volatile in both
directions) without zeroing each other out. Lisp-side =(set-mm-demand
NAME EUR)= / =(set-mm-surplus NAME EUR)= take effect on the next
refresh tick.

** Public order book stream semantics

Per the proto:

- One record per visible book event (add/modify/match/delete).
- Records may be batched per response.
- =update_time= drives replay range.
- Filter on (delivery_period, delivery_area, side).

Implementation: a single =tokio::sync::broadcast= channel of
=PublicBookEvent= covers the whole sim. Per-subscription filters live
in the stream task. Historical replay uses an in-memory ring (default
24 h, configurable) with timestamps; the stream task drains the ring up
to =now()=, then switches over to the live broadcast.

** Pre-trade validation matrix

| Rule                              | Error code on fail        |
|-----------------------------------|---------------------------|
| qty > 0 and multiple of step      | =VALIDATION_FAIL=         |
| price multiple of tick            | =VALIDATION_FAIL=         |
| currency matches area's default   | =VALIDATION_FAIL=         |
| delivery start aligned to duration| =VALIDATION_FAIL=         |
| period.start in (gate_open,close) | =VALIDATION_FAIL=         |
| area ∈ gridpool's allowed areas   | =VALIDATION_FAIL=         |
| valid_until in future or unset    | =VALIDATION_FAIL=         |
| ICEBERG: display_qty in (0, qty)  | =VALIDATION_FAIL=         |
| ICEBERG: peak_price_delta ≥ 0     | =VALIDATION_FAIL=         |
| STOP_LIMIT: stop_price set        | =VALIDATION_FAIL=         |

Rules live in =sim::order::validate(...)=. Server returns
=tonic::Status::invalid_argument= or yields a =FAILED= =OrderDetail=
with =StateDetail.state_reason=VALIDATION_FAIL=, mirroring the proto
notes.

;; --- runtime setup ---
(unless (boundp 'tradingsim-loaded)
  (setq tradingsim-loaded t)
  (load "sim/common.lisp")
  (load "sim/counterparties.lisp"))

(load "sim/defaults.lisp")
(watch-file "sim/defaults.lisp")

(reset-state)

(set-physics-tick-ms 100)
(set-clock-mode 'sim-time)            ; 'wall-clock or 'sim-time
(set-sim-time-multiplier 60.0)        ; 1 sim-min per real-sec
(set-sim-time-start "2026-05-13T08:00:00Z")
(set-socket-addr "[::1]:8810")
(set-ui-addr "127.0.0.1:8811")

;; --- markets ---
(make-market
 :area 'de-lu
 :code "10Y1001A1001A82H"
 :currency 'eur
 :tick 0.01
 :qty-step 0.1
 :durations '(60 15)                  ; 60-min + 15-min products
 :gate-open-h-before '((60 . 39)
                       (15 . 2))      ; 60-min opens 15:00 D-1 ≈ 39h
 :gate-close-min-before '((60 . 0)
                          (15 . 0)))

;; --- gridpools ---
(make-gridpool
 :id 1
 :name "battery-arb-pool"
 :areas '(de-lu)
 :tag-defaults '("strategy=arb"))

;; --- counterparties / background liquidity ---
(make-counterparty
 :name "mm-de-lu"
 :areas '(de-lu)
 :strategy 'market-maker              ; canned strategy from counterparties.lisp
 :params '((spread . 0.4)
           (size . 5.0)
           (refresh-ms . 1500)))

(make-counterparty
 :name "iceberg-pv-seller"
 :areas '(de-lu)
 :strategy 'renewable-iceberg-seller
 :params '((peak-mw . 25.0)
           (display-mw . 1.0)
           (peak-price-delta . 0.5)
           (target-price-fn 'pv-price-curve)))

;; --- price reference curves ---
(defun pv-price-curve (delivery-period)
  ;; midprice forecast that decays through the day
  (let ((h (delivery-hour-of-day delivery-period)))
    (cond ((< h 6) 95.0)
          ((< h 12) (- 95.0 (* (- h 6) 8.0)))
          ((< h 18) (+ 47.0 (* (- h 12) 6.0)))
          (t 83.0))))

;; --- scheduled animation ---
(every
 :seconds 60
 :call (lambda ()
         ;; nudge mid-price baseline by a small AR(1) noise
         (set-area-baseline 'de-lu
                            (+ (* 0.97 (area-baseline 'de-lu))
                               (* 0.03 95.0)
                               (- (/ (random 100) 25.0) 2.0)))))

(run-with-timer
 30  ; sim-seconds
 (lambda ()
   ;; one-shot scenario kick
   (start-scenario 'gate-closure-crunch)))
#+end_src

** Reload semantics

Same model as switchyard. =(reset-state)= drops the World, cancels
tulisp-async timers, and the next eval rebuilds. Counterparties and
their lisp closures are re-registered with their =:every= cadences.
Orders that were already on the book disappear on reload — they're not
intended to be durable. (A v0.4 option is =:preserve-book= for hot
config edits.)

* gRPC service implementation

All 11 RPCs of =ElectricityTradingService=. Each entry: =[scope]= what
state it touches, key validation, output shape, stream semantics.

** CreateGridpoolOrder
- Validate: order.delivery_area ∈ gridpool.areas; price/qty grid;
  duration/start alignment; FOK/IOC ⇒ no =valid_until=.
- Stamp =order_id= (monotonic), =create_time=, =modification_time=,
  set =OrderState=PENDING=.
- Push to admit-queue. Next tick: validate gate window, hand to the
  matcher, transition to ACTIVE / FILLED / CANCELED.
- Response: =OrderDetail= at first observable state. (FOK/IOC clients
  see the post-match state in the same call; LIMIT clients see ACTIVE
  with whatever immediate fills were synthesised.)

** UpdateGridpoolOrder
- Field mask gating on UpdateOrder fields per the proto comments
  ("not in update_mask: leave unchanged; present in payload: set;
  absent in payload: clear").
- Price/qty/exec changes that improve nothing keep priority; otherwise
  the order is removed and re-inserted at the tail.
- valid_until updates re-anchor the expiry tracker.
- Reject updates on terminal orders (FILLED/CANCELED/EXPIRED/FAILED)
  with =NOT_FOUND= (matches typical exchange behaviour).

** CancelGridpoolOrder / CancelAllGridpoolOrders
- Remove from book, transition to CANCELED with
  =STATE_REASON_DELETE=, =MARKET_ACTOR_USER=.
- =CancelAll=: walk the gridpool's order index, cancel every non-terminal
  order; one stream emission per order.

** GetGridpoolOrder
- O(1) lookup in the gridpool's order index. =NOT_FOUND= if missing.

** ListGridpoolOrders
- Filter: state(s), side, =DeliveryTimeFilter=, area, tag, ids.
- Pagination: cursor-based, opaque token = base64(=last_order_id=);
  page size capped (config; default 200, max 1000).

** ReceiveGridpoolOrdersStream
- Per-stream task: subscribe to the gridpool's broadcast topic;
  evaluate the filter; emit =ReceiveGridpoolOrdersStreamResponse=.
- Stream-lifetime contract per the proto: close when
  =delivery_time_filter.time_interval.end_time= has passed and the
  filter is exhausted (we know this because we know all contracts up
  to end_time and can tell when none of them have pending order
  state updates). Best-effort.

** ListGridpoolTrades / ReceiveGridpoolTradesStream
- Analogous to orders: filter on states/side/period/area/tag/ids.
- Stream-task subscribes to the per-gridpool =trade_broadcast= topic.

** ReceivePublicTradesStream
- replay window over =execution_time=, then live.
- Filter: states (always ACTIVE for v0.4; cancel/recall later),
  delivery_period, buy_area, sell_area.

** ReceivePublicOrderBookStream
- Replay window over =update_time=, then live. Filter: period, area,
  side. Records may be batched.

** Auth (HMAC-SHA256)
The proto notes auth on all RPCs is HMAC-SHA256 + key id in metadata.
For the simulator:
- =(set-auth-mode 'off | 'log-only | 'strict)= in lisp.
- =(define-auth-key ID SECRET …)= adds a key.
- =strict= mode rejects with =UNAUTHENTICATED= when missing/invalid.
- Default: =off= — most users want frictionless local sim.
v0.7+.

* Scenarios

Scenarios are an operator-facing layer for replaying typical-day
price patterns at chosen points along a 24-hour curve. They don't
write prices directly — they tilt aggressor side-bias so prices
drift via the MM's existing follow-last-trade machinery. The
"shape" of a day (when peaks land, how deep the belly goes) is
data; the prices are an emergent consequence of biased flow.

** Default behaviour — natural duck curve, always on

Without any scenario active, every aggressor's side-bias is
recomputed every 30 s from the hour-of-day of its target
contract's delivery start:

  - 00:00–05:00  bias 0.50  overnight, balanced
  - 05:00–09:00  bias 0.50 → 0.62  morning ramp
  - 09:00–10:00  bias 0.55  post-peak
  - 10:00–15:00  bias 0.35  solar belly (PV operators dumping)
  - 15:00–17:00  bias 0.50  transition
  - 17:00–21:00  bias 0.72  evening peak
  - 21:00–23:00  bias 0.60
  - 23:00–24:00  bias 0.50

The curve is the same shape for every quarter — q0's bias reflects
the very-imminent contract's hour, q47's reflects a contract
12 hours out. Prices emerge organically as those imbalances
collide with the MM's follow-last-trade pull.

** Scenario as a wallclock override

A scenario is a named partition of 00:00–24:00 into stages
(time windows). Each stage carries a target bias (or a gradient
within the window). When a scenario is active, every aggressor's
effective bias is

  bias_eff(i) = lerp(bias_natural(period_hour(i)),
                     bias_scenario(stage, when_within_stage),
                     decay(i))

where

  - =bias_natural= is the default duck-curve value for the
    contract's actual delivery hour. Always there.
  - =bias_scenario= is the scenario's current stage's target,
    optionally interpolated within the stage.
  - =decay(i) = exp(-i / 12)= ramps from 1.0 at q0 down to ~0.02
    at q47. The scenario's tilt fully applies to the
    next-quarter contract and fades over the next ~3 h. Long-dated
    contracts keep tracking the real wallclock.

Net effect: the orderbook shows prices "from a different time of
day" on the imminent contracts while the long-dated ones keep
tracking real wallclock. The actual delivery start times stay put.

** Stage spec

Each stage entry in the lisp =define-scenario= is a 5-list:

  (NAME HOUR-FROM HOUR-TO BIAS-FROM BIAS-TO)

  - =NAME= — UI label, conventionally "HH:00" or "HH:00 label".
  - =HOUR-FROM=, =HOUR-TO= — wallclock window the stage covers
    (float hours; 6.0 = 06:00, 17.5 = 17:30).
  - =BIAS-FROM=, =BIAS-TO= — target bias at the start and end of
    the window. Linearly interpolated between as the wallclock
    elapses (or as the stage is manually held).

Stages within a scenario must be ordered and contiguous,
partitioning [0, 24). The last stage's =HOUR-TO= is 24.0.

A flat-bias stage uses =BIAS-FROM = BIAS-TO=.

** Activation rules

POST =/api/scenarios/{name}/start=
  Pick the stage whose [HOUR-FROM, HOUR-TO) contains the current
  UTC hour. Set =current_stage= to that index. Clear
  =manual_override=.

POST =/api/scenarios/{name}/next= | =/prev= | =/jump/{idx}=
  Move =current_stage= to the requested index. Set
  =manual_override=. If the new stage matches the wallclock-current
  stage, clear =manual_override= instead.

POST =/api/scenarios/{name}/stop=
  Reset runtime. Aggressors fall back to the natural duck curve.

A 30-second tick task in the binary:
  1. For each running scenario without =manual_override=, recompute
     the wallclock-matching stage and set =current_stage= to it.
  2. Walk every aggressor; compute =bias_eff(i)= for its quarter
     offset; write the result into its =SharedConfig=.

The same tick handles both auto-advance and bias application — they
share the same lock window over the registry + aggressor map.

To resume wallclock-following from a manual jump: click the
wallclock-matching stage (server detects equality and clears
=manual_override=), or Stop + Start.

** Two scenario shapes coexist

The existing =elaborate.lisp= scenario uses an OPERATOR shape:
each stage is =(NAME FN-NAME)= and the UI invokes the named defun
when the operator clicks Start / Next. No wallclock binding.

The new shape is TIME-OF-DAY: stages are 5-lists with hour windows
and bias targets, no defun. The =define-scenario= defun
discriminates on the arity of the first stage entry — 2-element →
operator, 5-element → time-of-day. Stored as different
=Stage::Operator { fn_name }= vs =Stage::TimeOfDay { hour_from,
hour_to, bias_from, bias_to }= variants.

Auto-advance and bias-application only apply to TimeOfDay scenarios.
Operator scenarios behave exactly as today.

** Bias curve constants

Codified once in =scenarios.rs= (Rust):

  fn natural_duck_bias(hour: f64) -> f64 { /* piecewise as above */ }
  fn decay_weight(quarter_offset: i64) -> f64 {
      (-((quarter_offset as f64) / 12.0)).exp()
  }
  fn lerp(a: f64, b: f64, t: f64) -> f64 { a + (b - a) * t }
  fn stage_bias_at(stage: &TimeOfDayStage, hour: f64) -> f64 {
      let t = (hour - stage.hour_from) /
              (stage.hour_to - stage.hour_from);
      lerp(stage.bias_from, stage.bias_to, t.clamp(0.0, 1.0))
  }

Aggressor side_bias is the result; range clamped to [0, 1].

* Open design questions

** Q1. Determinism vs wall-clock?
Switchyard runs in wall-clock. For trading sims, users want both:
  - fast, deterministic replays for tests / ML training;
  - wall-clock or near-wall-clock for live demos against real bots.

The plan above gives both via =(set-clock-mode 'sim-time)= +
=(set-sim-time-multiplier X)=. Open question: do scheduled tulisp
=(every …)= timers fire on sim-time or wall-time? Switchyard's
tulisp-async runs against tokio's wall clock today. Option A: leave
=(every …)= as wall-time, expose =(every-sim …)= for sim-time.
Option B: rewrite tulisp-async timer source. A is cheap and the
common case.

** Q2. Iceberg slice price-step semantics?
The proto comment for =peak_price_delta= says "price difference
between the peak price and the limit price". The continuous-matching spec
treats it as the increment applied per slice. v0.2 implements
"each new slice is inserted at the next price level
=limit ± peak_price_delta=", which matches a common reading; the
intent is auditable in =matching.rs::insert_next_iceberg_slice=.
Worth re-validating against an actual production trace before v0.2 ships.

** Q3. Cross-zone coupling capacity granularity?
Real SIDC capacity is per-direction, per-15-min, per-coupling. v0.3
starts with a single =:capacity-mw= number; the next step is
=(set-coupling-schedule …)= taking a CSV-driven schedule, mirroring
the =csv-load= pattern.

** Q4. Should counterparties own their lisp closures, or are they
data?
Switchyard's pre-tick mailbox drain calls user-provided closures.
We can either (a) reuse exactly that mechanism for counterparties
(closure per counterparty, called every N ms), or (b) have
counterparties as data with a fixed Rust strategy chosen by symbol
and read params from an alist. Plan: support both; canned
strategies in =counterparties.lisp= are sugar over the raw closure
form.

** Q5. Where do the audit-grade order/trade logs live?
For real-money trading sims, users want a replayable log of every
order/trade with hash chaining. Out of scope for v0.1; planned for
v0.7 as =(set-audit-log PATH)= writing JSONL with HMAC-chained
records.

#+begin_src rust
use std::path::PathBuf;

fn main() -> Result<(), std::io::Error> {
    let proto_root = std::env::var("TRADINGSIM_PROTO_ROOT")
        .map(PathBuf::from)
        .unwrap_or_else(|_| PathBuf::from("submodules/frequenz-api-electricity-trading"));

    let api_proto = proto_root
        .join("proto/frequenz/api/electricity_trading/v1/electricity_trading.proto");
    let common_proto_root = proto_root.join("submodules/frequenz-api-common/proto");

    println!("cargo:rerun-if-env-changed=TRADINGSIM_PROTO_ROOT");
    println!("cargo:rerun-if-changed={}", api_proto.display());

    tonic_prost_build::configure()
        .disable_comments(["."])
        .include_file("proto_v1.rs")
        .compile_well_known_types(false)
        .compile_protos(
            &[api_proto.as_path()],
            &[proto_root.join("proto").as_path(), common_proto_root.as_path()],
        )
        .inspect_err(|e| eprintln!("Could not compile protobuf files. Error: {:?}", e))
}
